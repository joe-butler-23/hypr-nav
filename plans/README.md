## Direction options

- Package the project as a normal installable CLI once behavior stabilizes.
  Evidence: README currently asks users to `cargo build --release` and manually
  `sudo cp` three binaries. Trade-off: packaging is lower leverage than fixing
  runtime correctness first.
- Consider a declarative support matrix for terminal/multiplexer/editor layers.
  Evidence: README mentions extending to Zellij or other editors and
  `src/lib.rs` already centralizes detection logic. Trade-off: adding a matrix
  before the current paths are fully covered would add maintenance surface.

## Findings considered and rejected

- Replace all `/tmp` fallbacks: not planned because Hyprland sessions normally
  provide `XDG_RUNTIME_DIR`, the code refuses ambiguous Hypr sockets, and no
  concrete live exploit path was evidenced in this audit.
- Rewrite process discovery into a trait-based architecture: not planned now;
  the repo is only ~3.4K lines and the existing direct functions remain
  understandable. Characterization coverage is more valuable than abstraction.
- Add Zellij support immediately: directionally plausible, but no current
  code path, config, or tests show user demand stronger than the correctness
  issues above.
